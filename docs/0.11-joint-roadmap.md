## 3. 中期（Mid）— 约 2–6 周

**目标**：在契约稳定前提下，提升 Slim 模式稳健性、可观测性、列表/同步边界体验；**仍优先加性、不 bump wire**。

### M1. Sessions 截断可观测与可选加深（**双方轻量**）

| | |
|---|---|
| **意义** | `X-Complete=false` 时用户可能「少会话」却无感知；当前 limit=500 通常够，大实例需可见。 |
| **ocdroid 任务** | 1. UI/日志：`Complete=false` 时轻提示或 debug 横幅（非阻塞）。<br>2. **不**默认自动二次拉（协议无前向 cursor / start 非 offset）。<br>3. 若产品坚持「更大全集」：评估仅 roots 场景提高 limit 或文档说明上限。 |
| **slimapi 任务** | 1. 维持三头语义不变。<br>2. 若未来要「可分页全集」：**新设计**（cursor/total 等）→ **另开契约议题**（可能仍加性）。 |
| **配合点** | 任何分页语义变更必须双方签字；禁止客户端发明 offset。 |

### M2. Cold-start / resync 体验与负载（**ocdroid 为主**）

| | |
|---|---|
| **意义** | reconfigured + resync 幂等冷启动正确但可能重；大 allowlist 下 QPS/电量/列表闪烁。 |
| **任务** | 1. 度量：cold-start 四件套耗时、sessions 条数、q/p scope。<br>2. 评估 coalesce 是否再扩「短窗 debounce」（不丢 mid-stream 必达 resync）。<br>3. 非 focus catch-up 并发上限/可见行策略（契约允许的客户端策略）。 |
| **slimapi 配合** | 保持 `server.reconfigured` 与 `resync` **语义分层、无同因双发**；文档化「合法先后双发」窗口（已对齐）。 |
| **完成标准** | 有数据支撑的调优 PR；无回归 N1。 |

### M3. 展开路径硬化（**ocdroid 为主，slimapi 备援**）

| | |
|---|---|
| **意义** | v0.11.10 修了 placeholder；真实 part 路径、413 halve、部分 errors[] 仍有体验边角。 |
| **任务** | 1. 真机收集 Failed(null) 残差（log 已较全）。<br>2. UI：展开失败重试/文案区分 network vs residual。<br>3. 评估是否可去掉「首调 404 探测 RTT」在确认全网 G6 后（能力缓存）。 |
| **slimapi 配合** | 1. 维持 placeholder 规则稳定直到 backlog「去 placeholder」。<br>2. G6 错误码/413/503 行为不变；若改 envelope 字段须加性并通知。 |
| **backlog 联动** | slimapi「去 placeholder」落地后：ocdroid 可删 message-level 特判（保留防御性亦可）。 |

### M4. 可观测与错误码一致性（**双方**）

| | |
|---|---|
| **意义** | 联调与线上排障统一语言：`code` 优先，禁解析 `detail` 字符串。 |

### M5. 集成/模拟器 Slim 烟测固化（**ocdroid**）

| | |
|---|---|
| **意义** | 减少「只靠真机一次」；模拟器 + live sidecar 可重复。 |

## 4. 远期（Long）— 约 1–3 个季度 / 按需

**目标**：协议演进与架构债；**默认不启动**，需双方立项。

### L1. 去 `thin_placeholder_*`（**slimapi 主导 / ocdroid 跟随**）

| | |
|---|---|
| **意义** | 简化客户端分支；thin 始终带可对齐 part id 或明确「无可展开」。 |
| **slimapi 任务** | 设计：无可渲染 part 时是否空 parts + 标志位 / 合成稳定 id；兼容旧客户端。 |
| **ocdroid 任务** | 保留 `isThinPlaceholder` 防御至最低支持版本下线；主路径可简化。 |
| **配合** | **加性优先**；若破坏 thin 形状 → 必须 wire bump 讨论（双方默认抗拒 bump）。 |

### L2. Sessions 真·全集/游标协议（**双方立项**）

| | |
|---|---|
| **意义** | 超大实例下 500 根会话仍截断时的正确性。 |
| **前提** | 上游 opencode 能力；slimapi 是否暴露前向 cursor/total。 |
| **任务** | 新契约草案 → 评审 → 实现 → 客户端分页 UI。 |
| **注意** | 与现 `start` 水位语义正交；禁止 silently 复用 offset 幻想。 |

### L3. Feature discovery / 能力位（**慎重**）

| | |
|---|---|
| **意义** | schema 三键**不能**表达「有 reconfigured / 有 G6」；长期多版本并存需要能力探测。 |

| **纪律** | 不把 `clientMin/clientMax` 误用为能力位；wire 大版本才动 `X-Slimapi-Version`。 |

### L4. SSE 增量与实时性（**产品驱动**）

| | |

| **意义** | v1 明确丢 `text.delta` / part 增量；省流优先。若产品要「更实时」需 v2 级讨论。 |
| **范围** | 可能 bump 或并行流；**超出 0.11 线默认范围**。 |
| **配合** | 单独 RFC；不阻塞 0.11 收口。 |

## 5. 依赖与配合矩阵（摘要）

```
近  N1 真机回归 ─────────────── slimapi: 保持 rev F 实例 + 协助复现
    N2 文档对齐 ─────────────── 双方本仓
    N3 health 三键(可选) ────── slimapi: 无变更
    N4 slimapi patch tag ────── ocdroid: 文档一句
中  M1 Complete 可观测 ──────── 分页协议变更需立项
    M2 cold-start 负载 ──────── reconfigured/resync 语义稳定
    M3 展开硬化 ─────────────── placeholder backlog 联动
    M4 错误码 ──────────────── 新 code 双向文档
    M5 集成测 ──────────────── fixture 可选
远  L1 去 placeholder ───────── 契约设计 + 兼容
    L2 sessions 全集 ────────── 新契约
    L3 能力发现 ─────────────── 慎重
    L4 实时 SSE v2 ──────────── 产品 RFC
```

### 接口变更纪律（双方必须同意）

1. **Wire 默认 1**；破坏性变更先出迁移方案再谈 bump。  
2. 加性字段/头/事件：先改 slimapi `v1-contract` + `CLIENT_CHANGES`，再改 ocdroid，再联调。  
3. 客户端**禁止**把 `X-Complete` 当权威全集；**禁止** Ready=false 当权威空。  
4. 发版：ocdroid 仅 `./scripts/release.sh`；slimapi 用己方脚本/tag。  
5. 评审门控：高风险协议变更建议双方 ≥1 名 reviewer；实现后 `check.sh`（各仓）。
