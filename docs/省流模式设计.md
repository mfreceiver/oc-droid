## 0. 背景与决策记录

当前应用通过 SSE `/global/event` 接收所有 session 的全量事件（含每 token 的 `message.part.delta`），实测流量 **69MB/5min**（`TrafficLogger.kt:23` 注释）。服务端（1.17.11）**不支持**按 session 过滤（Issue #9650 `not_planned`）、无 per-session replay 端点、无 ETag/条件请求。详见 `docs/opencode-sse-api.md` §0。

经多轮分析（含 @oracle + @gpter + @dser 独立评审），核心结论：

1. **SSE firehose 是流量大头**，必须可关。
2. **`time.updated` 单独做主信号不可靠**——`MessageUpdated`/`PartUpdated` 事件不直接写 `time.updated`（`projector.ts:262/312`），agentic 多轮 tool run 会"批量完成、最后才触发 Session.Updated"。实测 4 session 有 1 个 `time.updated` 落后最后一条消息 33.5s。**作为辅信号可接受，但不能作为唯一变化探测**。
3. **`session/status`（busy/idle）是可靠的主信号**——busy→idle 跳变可靠表示 run 结束。但 tick=5s 采样可能漏掉两个 tick 之间完成并回到 idle 的短 run（gpter 致命#1）。
4. **因此必须加第三层：`probeLatestMessageId`（content probe）**——直接比对最新消息 id，不依赖 `time.updated` 和 status 跳变。**三方评审（oracle + gpter + dser）全部独立得出此结论。**

### 关键决策表

| 项 | 决策 |
|---|---|
| A（time.updated 风险） | 接受。但**不单靠 time.updated**，加 `probeLatestMessageId` 作权威兜底 |
| B+C（信号架构） | **三层**：① status 主驱动（busy→idle reload）② time.updated 辅（idle 态快速跳过，每 3 tick）③ content probe 权威兜底（idle 态每 N tick，直接比最新消息 id） |
| D（处理范围） | 只处理当前打开/在看的 session |
| E（permission/question） | **省流模式前台由 M1 轮询补足**：每 tick 顺带拉 `getPendingPermissions()` + `getPendingQuestions()`，弥补 SSE 关闭后前台断流（dser 🔴-3） |
| F（休眠提示） | 顶部轻提示，非弹窗。文案不暗示"无变化"可靠 |
| G（多 tab） | 只轮询当前活跃对话；切 tab 立即重置+跑完整流程。非当前 tab 不保证实时，但可借每 tick 的全量 status 轻量更新 busy badge |
| H（模式切换） | 只重载当前 session，**不清其他 session 缓存**；加 3s 防抖避免来回切反噬省流 |
| 多 session status（Point 3） | 所有模式共有，打开会话列表时 `GET /session` 刷新列表 + `GET /session/status` 补 busy 圆点，与省流轮询解耦 |
| ccEP 根因实测 | 不做，信任字段 |

### 服务端硬约束（1.17.11 实测）

- 无 `?sessionID=` SSE 过滤（官方 not_planned）
- 无 per-session SSE replay 端点（`/api/session/{id}/event` 1.17.11 不存在）
- 无 ETag / If-None-Match / Last-Modified
- `session.time.updated` 仅在结构性 Session.Updated 时 bump；不保证消息完成时 bump
- `GET /session/status` 返回全部 session 的 busy/idle/retry 映射，极轻量
- `GET /session/{id}/message?limit=1` 返回完整消息体（含 parts），约数百 B~数 KB

## 1. 省流轮询状态机（M1 核心）

**作用范围**：仅 `currentSessionId`（当前可见 tab）。tick = 5s。

### 1.1 每 tick 执行流程

```
每个 tick（5s）：
  ① GET /session/status         ← 1 次请求，含全部 session
  ② GET /permission              ← 省流模式前台补足（弥补 SSE 关闭后断流，dser 🔴-3）
  ③ GET /question               ← 同上

  取 currentSession 的 status，与上一 tick 的 lastStatus 比对：

  ┌─ status 请求失败 ─────────────────────────────────────────┐
  │  不改变 lastStatus；不累加 unchanged 计数器；             │
  │  连续失败 ≥ 3 次 → 显示连接错误 + tick 退避（5s→10s→20s→30s）│
  │  单次恢复即重置退避                                       │
  └───────────────────────────────────────────────────────────┘

  ┌─ status == busy 或 retry ─────────────────────────────────┐
  │  视为 active 态（gpter 致命#3：retry 也是 run 在进行）    │
  │  显示"生成中"（retry 时显示"运行出错"）（见 §1.2）       │
  │  不累加休眠计数器                                         │
  │  active→idle 跳变 → 触发 catchUp flow（§1.3）             │
  └───────────────────────────────────────────────────────────┘

  ┌─ status == idle ──────────────────────────────────────────┐
  │                                                           │
  │  [第 2 层] 每 3 个 tick 插入一次 time.updated 探测：      │
  │    GET /session/{id} → 比对 time.updated                  │
  │      变了  → 清计数器 + 进入 catchUp flow（§1.3）         │
  │      没变  → 不累加计数器（time.updated 不可靠，变化          │
  │              信号由 content probe 负责）                  │
  │                                                           │
  │  [第 3 层·权威兜底] 每 CONTENT_PROBE_INTERVAL 个 tick     │
  │  （默认 6 = 30s）插入一次 content probe：                 │
  │    probeLatestMessageId(sessionId) → 最新消息 id          │
  │    （OpenCodeRepository.kt:257，limit=1，通常 < 1KB）     │
  │    与本地 anchorNewestId（`maxByOrNull{time.created}`，    │
  │    非 `messages.last()`）比对：                            │
  │      相同  → 计数器 +1                                    │
  │              计数器 ≥ MAX_UNCHANGED → 顶部轻提示          │
  │              "省流模式下可能不是最新，点击检查"            │
  │      不同  → 清计数器 + 进入 catchUp flow（§1.3）         │
  │                                                           │
  │  （轻提示点击 = 手动刷新 = 清计数器 + catchUp flow）       │
  └───────────────────────────────────────────────────────────┘
```

### 1.2 retry 态处理

- `retry` 与 `busy` 同为 **active 态**，不累加休眠计数器。
- `(busy|retry)` → `idle` 跳变均触发 reload。
- `retry` 持续 ≥ 3 tick（15s）→ 顶部提示"运行出错，点击重试"（gpter 致命#3 / dser 🔴-2）。

### 1.3 catchUp flow（统一入口）

**触发源**：active→idle 跳变 / time.updated 变化 / content probe 命中 / 回前台 / 切 tab / 手动刷新 / 模式切换。

**流程**（复用并扩展 `launchCatchUp`，`MainViewModelSessionActions.kt:733`）：

```
1. anchorNewestId = messages.maxByOrNull{time.created}?.id（非 messages.last()）
2. probeLatestMessageId(sessionId)
3. 若 id 相同 → 跳过（零 reload 流量）
4. 若不同 → getMessagesPaged(id, 4, before=null)  ← 拉 4 条（limit+1 sentinel）
   + §preserveUnfetched merge（MainViewModelSessionActions.kt:563-599）：
     - 若第 4 条 id == anchorNewestId → 无 gap，anchor 刚好在 sentinel 位置
     - 若第 4 条 id ≠ anchorNewestId → gap 打开（gpter 致命#4：off-by-one 修正）
   → launchCloseGap（§1.4）
```

### 1.4 gap 闭合

复用 `launchCloseGap`（`MainViewModelSessionActions.kt:859`），改动：
- 首轮 sentinel 拉 4 条；gap 闭合从 tail cursor 继续，step = 3。
- 累计拉取 ≥ `GAP_CLOSE_MAX_STEPS`（默认 5）仍未闭合 → gap hint + "加载更多"按钮。
- 手动点击 → 继续 step=3 步进，预算重置。
- 当前 `launchCloseGap` 硬编码 `getMessagesPaged(..., limit = 5, ...)`（`:878`），需参数化支持 step=3。

### 1.5 计数器与休眠提示

- 计数器仅在 **idle + content probe 不变** 时累加。
- **active（busy/retry）态 + idle+time.updated 不变 均不累加**。
- 计数器 ≥ `MAX_UNCHANGED`（默认 5）→ 顶部轻提示"省流模式下可能不是最新，点击检查"。
- `isLoadingMessages=true` 时 tick 跳过，不推进计数（gpter 重要#1）。

### 1.6 退避

- 连续 6 轮（30s）idle 且无 reload → tick 拉长到 30s。
- 退避后 content probe 周期与 tick **解耦**：固定最多 30s 一次 probe，避免退避后变成 90s 才探（gpter 可选#1）。
- 用户发消息 / 手动刷新 / 回前台 / 切 tab → 重置回 5s。

### 1.7 切 tab（G）

`SessionSwitcher.switchTo`（`SessionSwitcher.kt:167`）切换 → 立即重置计数器 + 重置 tick 到 5s + 立即进入 catchUp flow（§1.3）。其他 tab 在切回去之前不轮询。

非当前 tab 的 busy/idle badge 可借每 tick 的全量 status 轻量更新，但**不拉消息**（gpter 重要#2）。

### 1.8 模式切换（H）

省流模式开/关 → 仅对当前 session 重载（resetLimit=true），**不清 `sessionWindowCache`（`SessionSwitcher.kt:105`）中其他 session 的缓存**（dser 🟠-3）。加 3s 防抖避免快速来回切换反噬省流。

### 1.9 省流模式下 sendMessage 交互（gpter 可选#3）

发送消息后：
- 本地立即标记 busy（`sessionStatuses[id] = "busy"`，不依赖轮询）。
- tick 保持 5s（不退避）。
- 后续轮询观测到 active→idle → catchUp flow 做权威 reload。

## 2. 省流模式场景表

> 仿 `docs/opencode-sse-api.md` §8.2 格式。省流模式下该 §8 全部场景假设（SSE 在线）不再适用，由本表取代。

## 4. delta coalescing 详细设计（M5）

> 非省流专属，独立改动。不影响省流模式（省流关 SSE，无 delta）。

### 4.1 合并策略

在 `SessionSyncCoordinator.kt:362-392` 的 `message.part.delta` 处理路径中：

1. **leading-edge immediate**：首个 delta 立即写入 `streamingPartTexts[partId]`（零延迟）。
2. **trailing coalesce**：后续同一 `partId` 的 delta 在 100ms 窗口内拼接，窗口到期一次性 flush。
3. **按 partId 分组**：不同 part 独立计时，互不干扰。

### 4.2 flush / 取消时机

- 收到 full `message.part.updated`（`:340-360`）→ 取消该 partId 所有 pending delta，以全量快照覆盖 `streamingPartTexts`。
- session 切换（`SessionSwitcher.switchTo`）→ flush 或丢弃新 session 不关心的 partId buffer。
- SSE stop（`ConnectionCoordinator.cancelSse`） / ViewModel clear → flush 残留 buffer。
- 100ms 窗口到期 → flush。

### 4.3 失序处理

- 按接收顺序拼接，不做排序。SSE 同 partId 的 delta 在同一 TCP 连接上保证顺序。

## 5. R-16 集成映射（v4 新增）

R-16 重构将大量逻辑从 `MainViewModel.kt` 抽到 `ui/controller/*`。本节说明各 controller 与省流模块的关系。

### 5.1 ForegroundCatchUpController vs M1（关键）

`ForegroundCatchUpController.kt` 是 **SSE 模式**的前台 catch-up 状态机：
- 依赖 `server.connected` 事件驱动 catch-up（`:174`）
- 三档：<15s throttle / 15s-5min SSE 重连 catch-up / >5min 冷启动（`:137-155`）
- 回调 `cancelSse()`/`forceReconnect()`/`catchUpAfterDisconnect()`

**省流模式下**：
- SSE 关闭 → 无 `server.connected` → ForegroundCatchUpController 的 catch-up 不触发
- 其 `cancelSse()`/`forceReconnect()` 回调天然 no-op（SSE 未启动）
- **M1（LowTrafficPoller）接管前台同步角色**：订阅 `AppLifecycleMonitor.isInForeground`，回前台时立即进入 catchUp flow（§1.3）

**集成方式**：M1 与 ForegroundCatchUpController **互斥并行**，按 `KEY_LOW_TRAFFIC_MODE` 激活其一。M1 启动时 ForegroundCatchUpController 的订阅仍在但无副作用（SSE 未启）。

### 5.2 其他 controller 关系

| Controller | 省流模式下的行为 |
|---|---|
| `ConnectionCoordinator` | `startSSE()` 被 M3 守卫跳过；`cancelSse` no-op |
| `SessionSyncCoordinator` | 省流模式无 SSE 事件，handler 不触发；M5 coalescing 仅非省流模式生效 |
| `SessionSwitcher` | 正常工作；M4 叠加省流 catchUp |
| `HostProfileController` | 不受影响 |

## 9. 风险登记

| 风险 | 等级 | 缓解 | 跟踪 |
|---|:--:|---|---|
| time.updated 某些消息类型不 bump（ccEP 反例） | 中 | **content probe 兜底**（P1 内置）；active→idle 主 reload | 上线监控 |
| tick 采样丢 busy→idle 跳变（短 run 在两 tick 间完成） | 中 | **content probe 兜底**——直接比消息 id，不依赖 status 跳变 | 上线监控 |
| 省流模式期间无 live token 流 | 低（用户已接受） | active→idle 后 catchUp flow 拉取，结果不丢 | — |
| 非当前 tab 无实时刷新 | 低（已接受） | 切回时 catchUp flow；每 tick 全量 status 可轻量更新 badge | — |
| permission/question 前台延迟（原 SSE 秒级变轮询 tick 级） | 低 | M1 每 tick 拉 permission/question，延迟 ≤5s | — |
| 关 SSE 后无法感知新 session 创建 | 低 | 打开会话列表时 `GET /session` 刷新发现 | — |
| M1 与 ForegroundCatchUpController 并行时的状态污染 | 低 | 省流模式 SSE 未启，ForegroundCatchUpController 回调 no-op；M1 独占前台同步 | 集成测试验证 |
