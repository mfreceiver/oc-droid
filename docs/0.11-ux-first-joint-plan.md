## 1. 术语表（m1）

| 术语 | 定义 |
|---|---|
| **success envelope** | HTTP 200 envelope：`items[]` 非空、`errors[]` 为空。全部请求 mid 均已成功交付。 |
| **partial** | HTTP 200 envelope：`items[]` 与 `errors[]` 均非空。部分 mid 成功交付，其余 mid 已被分类为可重试或终态 error。 |
| **errors-only envelope** | HTTP 200 envelope：`items[]` 为空、`errors[]` 非空。它不是 partial；客户端必须逐 error code 分类。若全部 errors 均为 mid-terminal code，则该响应也是 **terminal envelope completion**；若包含可重试 code，则仍有 unresolved mid，客户端仅重试这些 mid。 |
| **terminal envelope completion** | `errors-only envelope` 的子类：全部 `errors[]` 均为已知 mid-terminal code，因此请求集合已得到完整终态分类，不再自动重试。 |
| **请求级非 200** | 413 `response_too_large` / 503 `upstream_unavailable` / 503 `transform_busy`——请求级可重试响应，非某 mid 业务终态。 |
| **residual / unresolved** | 原请求 ids 中未成功 merge、未标 mid 终态者。预算耗尽/取消/进程死亡 → **自动恢复耗尽态**（保留 skeleton + 可点重试）。 |
| **mid-terminal code** | envelope errors[] 中 `message_not_found` / `message_too_large`——mid 自动展开不可完成，保留 skeleton。 |
| **自动恢复耗尽态** | 预算/取消/进程死亡时 residual 状态：retry affordance + skeleton 保留；非业务终态。进程死亡不持久化——重启后 skeleton 默认可展开态恢复 affordance。 |
| **attempt**（三种，不共享） | ① 逻辑操作（恒 1）/ ② 瞬态重试（503·网络·超时·envelope 可重试 code）/ ③ 分区请求（413 拆批）。 |

#### R4-B2-RESULT-MATRIX（完整响应矩阵 · rev-bgpt 四次复审成稿）

rev 4「零成功 item→一律 503」矫枉过正已修；rev-bgpt 四次复审指出残余分类缺口（全成功被误称 partial / errors-only 术语冲突 / 漏 mid-retryable errors），并给出完整成稿（本节即采用其文本）。**错误类别（矩阵用）**：

- **network error**：mid 读取发生 `httpx.RequestError`；仅 Opt-A 分支将其映射为 envelope 可重试 code。
- **mid-retryable error**：已形成 envelope error、但客户端可 bounded retry 的 code，如 `upstream_http_5xx`、429，以及 Opt-A 映射出的 network error code。
- **mid-terminal error**：客户端不得自动重试的已知终态 code，如 `message_not_found`、`message_too_large` 和按契约分类为终态的 4xx。
- **unknown error code**：客户端 fail-open 到自动恢复耗尽态，保留 skeleton 与 retry affordance。

**完整响应矩阵**：

| 请求能力 | 成功 items | RequestError network mids | 其它 envelope errors | 响应 |
|---|---:|---:|---|---|
| **非 opt-in** | 任意 | 任意 | 任意 | **完整执行 legacy 分支**：不得先应用 Opt-A 映射；响应状态、body 与错误优先级保持部署前语义（R4-B2-OLD-SEMANTICS）。 |
| opt-in | ≥1 | 0 | 0 | **200 success envelope**：`items[]` 非空，`errors[]` 为空。 |
| opt-in | ≥1 | 任意 | 任意非空 | **200 partial**：成功 mid 进 `items[]`；network、mid-retryable、mid-terminal errors 均进 `errors[]`。客户端立即 merge items，仅重试可重试 errors。 |
| opt-in | 0 | 0 | ≥1 | **200 errors-only envelope**：保持已有 per-mid HTTP error 语义。客户端按 code 分类；若全为 mid-terminal code → terminal envelope completion；若含 `upstream_http_5xx`/429 等 mid-retryable code → 仅 bounded retry 这些 mids。 |
| opt-in | 0 | ≥1 | ≥1 | **200 errors-only envelope**：RequestError mids 映射为可重试 code，其它 errors 保持原 code；客户端仅 bounded retry network/mid-retryable mids，mid-terminal mids 不重试。 |
| opt-in | 0 | ≥1（覆盖全部请求 IDs） | 0 | **顶层 503 `upstream_unavailable`**：所有请求 IDs 均 unresolved，客户端按顶层 503 预算整批重试。 |

**Opt-A 响应规则提炼**：

1. 非 opt-in 请求**必须在 RequestError→envelope 映射之前进入 legacy 分支**，逐场景保持部署前语义。
2. opt-in 请求只要存在至少一个可交付或可分类的确定结果（成功 item，或非 RequestError 形成的 envelope error），就返回 HTTP 200 envelope。
3. 仅当全部请求 IDs 都因 RequestError 失败、无任何成功 item、也无任何其它 envelope error 时，返回顶层 503。
4. HTTP 200 ≠ 所有 mids 均成功或终态；客户端必须按 `items[]` 和每个 `errors[].code` 分别 merge、retry 或标终态。
5. `partial` 仅指 `items[]` 与 `errors[]` 均非空；errors-only 响应统一称 `errors-only envelope`（§1）。

**C1 澄清（累计 413 × opt-in · slimapi 联审共识）**：Opt-A 变更面 = **仅 mid `httpx.RequestError` 映射为 envelope 可重试 code**。**累计字节超限 `response_too_large`（顶层整请求 413）对 opt-in / 非 opt-in 一致**——保持顶层 413、succeeded 不输出、B1 分区恢复统一适用（不返 200 partial）。per-mid `message_too_large` 与 mid HTTP≥400 同理对 opt-in/非 opt-in 一致。理由：413 = batch 过大 → 确定性分区（B1）；Opt-A = 网络瞬态 → 保 partial 防重下，失败模式与恢复策略不同；保持 413 顶层使 B1 分区契约对 opt-in/非 opt-in 统一，最小变更面，且不重开 rev-bgpt 已 CLOSED 的 B1 契约（「顶层 413 不返 partial」）。**slimapi 与 ocdroid 双方确认，不改变矩阵语义，无需 rev-bgpt 复审。**

#### R4-B2-OLD-SEMANTICS（非 opt-in 逐场景不变 + 回归测试）

MockWebServer 钉 **非 opt-in 回归矩阵**（请求不带能力头）：
- 纯终态 errors（全 message_not_found）→ 200 errors-only（**不被误升 503**）
- success + network failure → 503（丢 success，旧行为，不改）
- 纯 network failure → 503
- 纯 HTTP errors（upstream_http_4xx 终态）→ 200 errors-only
- 全成功 → 200 success envelope（items[] 非空、errors[] 空）

**非 opt-in legacy 等价验收（I-R5-LEGACY-EQUIVALENCE，规范化语义等价，非逐字节）**：对固定 fixture，Opt-A 部署前与部署后非 opt-in 响应须满足——HTTP status 相同；`Retry-After` presence 与规范化值相同；JSON 解析后结构等价（code/items/errors/messageID/error code 全同）；items/errors 存在性 + 每数组按 messageID 规范化排序后内容相同；顶层 503 路径不得泄露 Opt-A partial items/errors；**忽略** Date/Server/trace-id 等非契约动态 header 与 JSON object key 顺序。不得仅因序列化空白、key 顺序或动态诊断 header 不同判 legacy 回归。

#### R3-B2-UNKNOWN（CLOSED）/ R3-B2-INVARIANT（CLOSED，rev 4）

未知 code → fail-open 自动恢复耗尽态。envelope 内 items/errors 按 messageID 互斥、幂等 merge、顺序无关；请求级非 200 下全部 IDs 返回 unresolved。

#### Opt-A feature flag + 回滚（I-R4-ROLLBACK-THRESHOLD）

服务端 Opt-A feature flag 守护；**回滚阈值（I-R4-ROLLBACK-THRESHOLD + I-R5-ROLLBACK-BASELINE）**：基线 = Opt-A 部署前最近 24h 同 endpoint、同客户端版本分层。观察窗 1h（S-A 指标），样本数须 ≥100 才触发自动关闭——① 「envelope 相关 5xx > 2× 基线」：基线率 >0 时 `currentRate > 2×baselineRate` 且样本 ≥100 触发；**基线率 =0 时** current 1h 出现率 >1% 且样本 ≥100 触发；② unknown-code 占比 >5%（样本 ≥100）；③ 客户端上报展开失败率 >5%（样本 ≥100）。样本不足只告警不自动关；允许责任人依据严重单例人工关闭。关闭后：后续请求（即使 opt-in）回原 503 语义；**进行中 operation 已 merge 的 partial items 不撤回**。

**wire**：保持 1（能力头非 bump）；bump 不排除（仅当能力头方案部署证明不足）。
