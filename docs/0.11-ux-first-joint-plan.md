# ocdroid × oc-slimapi — 体验优先联合方案（v0.11.10 / slimapi v0.3.0）

> **状态**：体验优先修订稿 · 待双方确认  
> **日期**：2026-07-21  
> **指导思想**：**尽快完善 App 体验**——避免「展开失败」、提升省流效果、改善弱网体验。  
> **Wire**：`X-Slimapi-Version: 1`（默认不 bump）  
> **基线**：ocdroid `v0.11.10` / `69724a2` 已发；slimapi `v0.3.0` / rev F 已部署（以探针为准）

---

## 0. 一句话

**契约主线已闭环，下一阶段不再以「文档/版本对齐」为主目标。**  
下一阶段以用户可感知结果排序：

1. **展开可靠**（无莫名「展开失败」、瞬态自动恢复、终态可理解）  
2. **弱网先展示缓存**（不清空、少闪烁、少请求风暴）  
3. **实测省流**（消灭冗余 full/cold-start，再谈更深 skeleton）  
4. **协议演进仅在数据证明后**（reasoning 省略、G6 部分成功语义等）

## 1. 产品目标（可验收）

### 1.1 展开

| 目标 | 说明 |
|---|---|
| 无莫名死胡同 | 合法可展开消息不应长期停在红色「展开失败」而无原因/无重试 |
| 瞬态自动恢复 | `503` / 超时 / 网络中断后，有限次数自动重试后应成功或进入明确可点重试态 |
| 部分成功保留 | 一批多 mid 时，成功的 mid 立即呈现；失败 mid 单独处理 |
| 413 完整处理 | 聚合过大时**两边 half 都处理**（当前实现只处理前半，是体验/正确性缺口） |
| 精确终态 | `message_not_found` / `message_too_large` / 离线耗尽 文案与行为分离，**不清掉已有 skeleton** |

> 绝对 100% 成功不可承诺（消息已删、硬尺寸上限）。可达成目标是：**没有无解释死胡同**。

### 1.2 弱网

| 目标 | 说明 |
|---|---|
| 缓存优先 | 502/503/超时/`Ready=false` 时**保留**会话/消息/q/p，不全屏清空 |
| Complete=false | 不完整页**不得删除**本页未返回的旧 session |
| 连接态与内容分离 | 「可能过期 / 重连中」轻量指示 + 上次成功时间；禁止重复 toast 轰炸 |
| 冷启动有界 | 同 generation 最多 1 个 in-flight cold-start；允许 1 次 trailing；无 stale generation 写入 |
| 手动优先 | 用户点展开/手动刷新**绕过**任何后台熔断 |

### 1.3 省流

| 目标 | 说明 |
|---|---|
| 零无故 full | 普通渲染只 skeleton；full 仅用户展开 |
| 零重复已加载 | 已 Loaded 的 mid 不在重试中再拉 |
| 冷启动不双拉 | 每 generation 正常 1 轮；connected+resync 竞争 ≤2 轮且不并行 |
| 实测字节 | 用现有 `TrafficCountingInterceptor` 扩展逻辑操作维度，再决定是否省略 reasoning 等 |

### 已知代码缺口（驱动近期）

| 缺口 | 位置/现象 | 体验影响 |
|---|---|---|
| G6 **413 只 retry 前半 ids** | `OpenCodeRepository.expandBatchInternal` ~2304 | 后半 mid 直接 residual 失败 → 用户「展开失败」 |
| 传输/超时 → `Failed(null)` 无自动重试 | 同文件 IOException 分支 | 弱网点一次就红 |

| 旧实例 404 每批都可能探测 G6 | `thin_route_not_found` fallback | 浪费 1 RTT/批 |

## 3. 近期 P0（体验主线，建议 1 个 ocdroid patch 周期）

### P0-A — 展开恢复状态机（**ocdroid 主导 · 最高优先**）

**工作包 O-A：Expansion recovery**

1. **逻辑操作级**展开（非裸 HTTP 次数）：
   - 去重 mid；一次用户动作 = 一个 operation budget（建议自动尝试 ≤4）。
2. **瞬态可重试**：
   - `503 transform_busy` / `503 upstream_unavailable` / `IOException` / timeout  
   - 优先 `Retry-After`（有上限），否则指数退避 + 抖动  
   - **禁止** 503 后扇出 N 路单条（放大过载）
3. **413 聚合**：
   - `response_too_large`：**递归处理前后两半**，合并 items/failedIds  
   - `message_too_large`：该 mid 终态「过大」，非红笼统失败
4. **部分成功**：
   - 200 + `errors[]`：成功 mid 立即 merge；仅对失败 mid 重试（若可重试）
5. **404 能力缓存**：
   - 每 host 记录「G6 不可用」→ 后续直走单条，不再每次探 batch
6. **同 mid 合并**：
   - 同 `(host, sessionId, messageId)` 同时最多 1 个展开 in-flight
7. **终态分类**（UI 文案）：
   - Loaded  
   - Retrying / Offline（可自动或手动）  
   - 不可用：已删除 / 过大 / 未知（带重试，**保留 skeleton**）

**验收（MockWebServer + 虚拟时间）**

| 指标 | 目标 |
|---|---|
| 健康 fixture 合法 mid | 100% Loaded |
| 注入 1–2 次 503/超时后成功 | 100% 最终 Loaded |
| 完成后仍 Loading | 0 |
| 413 两边 half 都尝试 | 100% ids 有结局 |
| 重试中再拉已 Loaded id | 0 |
| 同 mid 并发请求 | ≤1 |
| 单次用户动作自动尝试 | ≤4 |
| 已知错误仍显示笼统「展开失败」 | 0 |

### P0-B — 弱网缓存优先（**ocdroid 主导**）

**工作包 O-C**

1. `Complete=false`：**保留**该 directory 本页未返回的 prior sessions（单测钉死）  
2. 502/503/超时/Ready=false：**不清空**已有列表与聊天窗口  
3. 连接态与内容分离：轻量 stale/reconnecting 指示  
4. 冷启动：已有 coalesce 保持；补「无空列表闪烁」回归  
5. 用户点展开始终允许探测（后台 breaker 若有则 bypass）

**验收**

| 指标 | 目标 |
|---|---|
| 超时/503 后本地内容保留 | 100% |
| Complete=false 误删 prior session | 0 |
| 并发 cold-start | ≤1 |
| stale generation 提交 | 0 |

### P0-C — 省流止血（**ocdroid 主导 · slimapi 配探针**）

**工作包 O-D**

1. 扩展 `TrafficCountingInterceptor` 关联：operationId / expand vs cold-start / attempt / mid 集合  
2. 强制：普通路径无 full；重试仅 unresolved mids；每 generation 1 正常 cold-start 轮  
3. **F-1 运行时确认**（联合 fixture）：避免 watermark 循环 → 漏消息 **且** 重复流量  

**验收**

| 指标 | 目标 |
|---|---|
| 未展开前 full 请求 | 0 |
| 每 generation 正常 cold-start 轮 | 1（竞争 ≤2 且不重叠） |
| 旧 G6 404 探测 | ≤1 次/host·进程 |
| F-1 等时间戳 tuple 推进 | 无循环、无漏边界 |

### P0-D — 确定性故障矩阵（发版门）

在 `v0.11.11`（或下一 patch）前用 MockWebServer 钉：

- partial G6 errors  
- 503×2 后成功 + Retry-After  
- timeout 后成功  
- 413 需处理两半  
- mid 过大 / not_found 终态  
- 展开中断网  
- cold-start 中重连  
- Complete=false + prior 省略 sessions  

**契约/部署探针**（R3/R8/版本身份）改为**证据附件**，不再作为本阶段主里程碑。

## 4. 中期 P1（省流加深 · 数据驱动）

### M1 — cold-start 并行取件（ocdroid）

`coldStartSlimSync` 现串行；评估 `supervisorScope` 有界并发：

- 单件失败不拖死整轮  
- 仍 token/generation 门禁  
- 不因单件失败自动再开第二轮  

### M2 — reasoning 等大 part 省略（**联合 · 数据后启动**）

slimapi skeleton 现保留全文 reasoning（`skeleton.py` 附近）。先量字节占比：

- 若 reasoning 主导 → slim 侧 preview + `hasFull/omitted`  
- ocdroid 走已加固 G6 展开  
- **不默认省略** 普通 user/assistant 正文  
- 可仍 wire=1；需最低客户端版本说明  

### M3 — G6 部分成功 wire（联合 · 仅当多 mid 批量常见）

若指标显示多 mid 批次多，且 mid 级网络失败现升整请求 503 丢成功 items：

- 成功 items 保留  
- 失败 mid 进 `errors[]` 可重试 code  
- 先客户端兼容，再契约加性说明  

### M4 — 仅后台读请求熔断（可选）

仅当出现请求风暴：host 级 background-GET breaker；**手动展开/刷新绕过**。

## 6. 与 slimapi 的配合与协商

### 6.1 近期 slimapi（支撑体验，非新协议）

| ID | 任务 | 意义 |
|---|---|---|
| S-A | G6 指标：ids 分布、items/errors、413/503、latency、骨架字节、Retry-After 次数 | 驱动客户端退避与是否改 wire |
| S-B | 瞬态 503 **尽量带 Retry-After**（不仅 transform_busy） | 弱网展开成功率 |
| S-C | 确认 gzip thin 成功路径；按 part 类型粗粒度字节比 | 省流方向 |
| S-D | 提供 fixture：placeholder、等 timestamp 双 mid、discovery 翻转 | F-1 / 展开回归 |
| S-E | 部署身份可审计（tag/commit，避免 editable 漂移） | 联调不测错版本 |
| S-F | 保持 reconfigured/resync 分层；失败 discovery 不发假 reconfigured | 弱网不风暴 |

**近期 slimapi 不做**：去 placeholder、roots 默认、cursor、replay、健康三键当 feature flag。

### 6.2 必须协商（D 表 · 体验导向）

| # | 议题 | 建议默认 |
|---|---|---|
| **U1** | 瞬态 G6 code 表（可重试 vs 终态）+ Retry-After 语义 | 503/网络可重试；not_found/too_large 终态；客户端总预算 ≤4 次/≤10s |
| **U2** | 413 两半处理为客户端 MUST（契约文档可加一句） | **是** |
| **U3** | 是否推进「mid 网络失败不丢成功 items」 | **等字节与 batch 分布数据** |
| **U4** | reasoning 省略是否进下一 minor 联合窗口 | **测后再定**；正文不省略 |
| **U5** | 硬尺寸 too_large 是否 v1 终态 | **是**；新端点另议 |
| **U6** | 共享 2–3 个代表会话 fixture 测展开率与省流比 | **必须** |

原 D1–D7 中：

- Complete 不强制翻页 → **保留**，但补 **incomplete 不删 prior**  
- 去 placeholder → **降级**，不进本周期  
- F-1 → **保留为 P0 证据**（服务正确性 + 省流）  
- health 三键 UI → **延后**  
- roots 默认 → **不做**
